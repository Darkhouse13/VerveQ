# Contributing to VerveQ Platform

Thank you for your interest in contributing to the VerveQ Platform! We welcome contributions from the community to help improve and expand this competitive sports gaming platform.

VerveQ is a **React 18 + Vite frontend on a Convex backend**, written in TypeScript. There is no Python and no separate API server. [`README.md`](../README.md) is the source of truth for the stack and setup; this document covers the contribution process.

## Code of Conduct

By participating in this project, you agree to abide by our [Code of Conduct](CODE_OF_CONDUCT.md). Please treat all contributors and users with respect and consideration.

## How to Contribute

### Reporting Bugs

If you find a bug in the platform, please create an issue on GitHub with the following information:

1. A clear and descriptive title
2. Steps to reproduce the issue
3. Expected behavior vs. actual behavior
4. Screenshots or code examples, if applicable
5. Your environment information (OS, browser, etc.)

### Suggesting Enhancements

We welcome ideas for new features or improvements to existing functionality. To suggest an enhancement:

1. Check if there's already an existing issue or discussion
2. Create a new issue with a clear title and detailed description
3. Explain why this enhancement would be beneficial
4. Include any relevant examples or mockups

### Code Contributions

#### Getting Started

1. Fork the repository
2. Clone your fork locally
3. Create a new branch for your feature or bug fix
4. Make your changes
5. Add or update tests as necessary
6. Ensure all tests pass
7. Commit your changes with a clear, descriptive message
8. Push to your fork
9. Create a pull request

#### Development Setup

Setup lives in [README.md](../README.md) — "Local development". In short:

```bash
cd app
npm install
npx convex dev
npx vite
```

The Convex target resolves from `app/.env.local` (gitignored); you need `CONVEX_DEPLOYMENT` and `VITE_CONVEX_URL`. Server-side auth/email variables are set on the Convex dashboard, not in `.env.local` — see [AUTH.md](AUTH.md).

#### The gate

From `app/`, one command has to pass before you push:

```bash
npm run check
```

That is `codegen` + `tsc -b` + `lint` + `vitest` + `build` (`app/package.json`). CI runs the same command on every push and every pull request via [`.github/workflows/check.yml`](../.github/workflows/check.yml), so a red gate locally is a red gate on your PR.

The typecheck is a **hard gate** — master typechecks clean, so any new `tsc` error is a regression. Fail closed on it rather than working around it.

#### Pull Request Process

1. Ensure `npm run check` passes from `app/`
2. Update documentation to reflect your changes
3. Add tests for new functionality
4. Write a clear, descriptive pull request message
5. Link any related issues in your pull request

**Merging to `master` is a production release.** There is no confirmation step: [`deploy.yml`](../.github/workflows/deploy.yml) deploys the Convex backend — functions *and* schema — to production and then republishes the frontend. Read [DEPLOYMENT.md](DEPLOYMENT.md) before merging anything with a schema or Convex API change.

#### Coding Standards

- TypeScript throughout; the repo is ESM (`"type": "module"`)
- ESLint is the standard, enforced by `npm run lint` inside the gate
- Use meaningful variable and function names
- Write clear, concise comments for complex logic
- Keep functions and modules focused on a single responsibility
- Never hand-edit `app/convex/_generated/` — it is gitignored build output regenerated by `npm run codegen`

#### Testing

- All new code should include appropriate tests
- Tests are [Vitest](https://vitest.dev) (`npm run test` from `app/`, or `npm run test:watch`); they live alongside the code and in `app/src/test/`
- Write unit tests for business logic and contract tests for Convex functions
- There is no coverage threshold — the bar is that the gate passes and new logic is covered, not a percentage

## Development Guidelines

### Project Structure

| Path | Purpose |
| --- | --- |
| `app/` | The whole application — React frontend (`app/src/`), Convex backend (`app/convex/`), bundled runtime data (`app/convex/data/`), and build/QA tooling (`app/scripts/`) |
| `scripts/` | Data pipeline (fetch/ingest/build/seed) and curated-parity tooling |
| `docs/` | Documentation — freshness varies; see the docs status table in README.md |
| `deploy/` | Dockerfile, nginx config, and the image-based publish scripts CI drives |
| `.github/` | CI workflows (`check.yml` verifies, `deploy.yml` ships) |

### Backend Development

The backend is Convex functions in `app/convex/`: queries, mutations, and actions over the tables defined in `app/convex/schema.ts`.

- Game logic is **server-authoritative** — never trust the client for scoring or answer validation
- Shared helpers live in `app/convex/lib/`; reuse them rather than duplicating scoring, ELO, or fuzzy-matching logic
- Schema changes ship to production on merge. Rehearse them on the dev deployment first (see README.md, "Runtime shape")

### Frontend Development

- React 18 function components with hooks; React Context for cross-cutting state (see `app/src/contexts/`)
- React Router for routing; the production surface is the v2 shell (`app/src/pages/shell/`)
- Tailwind CSS for styling
- User-facing copy must be translated — the app ships English, Spanish, and French at exact key parity
- Ensure responsive design for all screen sizes
