# VerveQ

VerveQ is a trivia platform — football at its core, now spanning general-knowledge subjects — built on React 18 + Vite with a Convex backend. It is live at [verveq.com](https://verveq.com).

The production surface is the **v2 shell** (`VITE_V2_SHELL_ENABLED=true`, baked at deploy time). Legacy v1 screens are kept behind redirects as a rollback seam and never render in production.

## What's in the app

### Play (solo)

| Mode | Scope | Notes |
| --- | --- | --- |
| Quiz | Football | Ranked and casual paths via the Compete tab |
| Blitz | Football | Timed sprint, shared question engine with Quiz |
| Survival | Football, basketball, tennis | Initials-based, server-authoritative |
| Higher or Lower | Football | Difficulty tiers at mode select |
| VerveGrid | Football | Difficulty tiers; curated boards |
| Who Am I | Football | Approved-clue ladder |
| Daily | Football | One challenge per day; World Cup edition live. Quiz-mode only — daily *survival* is stubbed (`app/convex/dailyChallenge.ts`) |

### Compete (multiplayer)

| Mode | Notes |
| --- | --- |
| Duels | Async head-to-head via share links; rivalry ledger; OG share cards |
| Challenge Arena | Room-code MCQ rounds. Subjects: football, tennis, basketball, general knowledge, science & nature, world history, movies & TV, music, video games. Customizable rounds |
| Live Match | Real-time 1v1 (full accounts) |
| Ranks | ELO ladder with seasons and decay warnings |

### Learn

Hand-authored teaching ladders (21 modules across geography, history, science and more) with runner, review, and mastery loops — `app/src/pages/shell/learn/`, content in `app/convex/learn*LadderV1.ts`.

### Forge

Community question submissions with voting (`/v2/forge`).

The app ships in **English, Spanish, and French** (namespaces at exact key parity). Sentry is wired for production error reporting (errors-only, PII scrubbed).

## Runtime shape

- `app/convex/` is the live backend surface: ~28 tables, 8 cron jobs (season roll, daily generation, ELO decay, stale-match reaping, cleanups, ops email).
- `app/convex/_generated/` is gitignored build output regenerated by `npm run codegen`; do not hand-edit or commit it.
- Curated modes read approved layers, not raw pipeline tables: Higher or Lower reads `higherLowerPools`/`higherLowerFacts`, VerveGrid reads `verveGridBoards`, Who Am I reads `whoAmIApprovedClues`. Raw `gridIndex`/`statFacts`/`whoAmIClues` are pipeline/audit artifacts.
- Survival runs from bundled JSON datasets in `app/convex/data/` (not DB tables).
- The production Convex deployment is `different-lynx-153` (the project's prod slot). `admired-warthog-495.eu-west-1` is the dev/staging deployment — schema changes and destructive operations should be rehearsed there first.

## Local development

```bash
cd app
npm install
npx convex dev
npx vite
```

The Convex target resolves from `app/.env.local` (gitignored — you need `CONVEX_DEPLOYMENT` and `VITE_CONVEX_URL`). To validate a production bundle locally:

```bash
cd app
npm run build
npx serve -s dist -l 3000
```

## Code gate

From `app/`, `npm run check` is the full pre-push gate: codegen + `npx tsc -b` + lint + tests + build. The typecheck is a hard gate — master typechecks clean (0 errors; the old ~31-error baseline was eliminated 2026-06), so any new tsc error is a regression. Fail closed on it.

CI runs this same gate on every push and PR (`.github/workflows/check.yml`).

## Deployment

**Deployment is fully automated.** Every push to `master`:

1. `check.yml` runs the full code gate.
2. `deploy.yml` deploys the Convex backend (`npx convex deploy`, gated on the `CONVEX_DEPLOY_KEY` secret, which is configured) and publishes the frontend over SSH — a forced command on the host runs the image-based publish (`deploy/build-and-run.sh`) with the previous container retained for instant rollback and a `/healthz` check after cutover.

There is no manual deploy step in the normal flow. `docs/DEPLOYMENT.md` describes the underlying host topology and rollback procedure but predates the automation — where they disagree, the workflows are the truth.

## Curated runtime parity

Destructive curated parity fails closed by default and is approval-gated outside the repo (Windows DPAPI / macOS Keychain trust anchor; Linux and CI stay blocked).

```bash
cd app
npm run gameplay:curated-parity:status    # read-only self-check: target, allowlist, trust anchor, artifacts
npm run gameplay:curated-parity:inspect   # + current curated manifest
npm run gameplay:curated-parity:approve   # single-use approval (requires allowlisted target)
npm run gameplay:curated-parity           # destructive apply: authoritative reseed from scripts/data/*
npm run gameplay:smoke -- all             # verify parity + runtime startup afterward
```

Approval requires the exact target in the gitignored `.ops/curated-parity/approved-targets.local.json`; each approval is consumed by the first destructive step. Status state names: `localAllowlist` (`missing|malformed|matched|unmatched`), `trustAnchorStatus` (`available|missing|unsupported|unusable`), `approvalArtifactStatus` (`missing|valid|expired|mismatch|invalid|consumed`), `applySessionStatus` (`missing|active|expired|invalid`).

## Auth

Convex Auth with a **Password** provider (email + password, **8–72 chars**, common-leaked-password rejection, OTP password reset via Resend) plus an **Anonymous** provider for guest play (IP-permit rate-limited). Username-only onboarding with a later upgrade path to a full account. Flows live in `app/src/pages/LoginScreen.tsx` and `app/src/contexts/AuthContext.tsx`; policy in `app/convex/lib/passwordPolicy.ts`.

Required Convex dashboard env vars: `RESEND_API_KEY`, `EMAIL_FROM`, `CONVEX_SITE_URL`. See [`docs/AUTH.md`](docs/AUTH.md).
